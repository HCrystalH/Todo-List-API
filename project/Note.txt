Phase 3: Security and Authentication - The heavy one
We need to implement JWT authentication -> allows user log in once -> get a token -> show token for every subsequent request
We need to create 3 main components
    1. Security Configuration: To tell Spring which URLs are public (like /register and /login) which one require a token
    2. JWT utils: to handle creating the token and reading
    3. Authentication: handle the actual logic of checking password & return token
        * add JWT dependency to pom.xml
        * hashing password
    4. Data Transfer Layer (DTOs)
        * Never send password back to user in JSON response, then customize response and request format
    5. Some noted points:
        - When a task need connection to Database -> we need @Transactional
        - meta-annotated with @Component
            + @Component: use cases -> Generic bean ( Generic utilities)
            + @Repository : use cases -> Data access layer ( Database layer)
            + @Service : use case -> Business logic layer
            + @Controller: use case -> Web layer (MVC)  ( APIs)
        - @Bean , Bean scope, bean lifecycle
        - @Component vs @Configuration
        - hibernate ddl-auto, there are 2 differences in create and update mode'
        - JWT is stateless and that's why we dont store it in Database --> no cost because no query

Phase 4: Todo CRUD & Authorization (building the endpoints and ensuring users can only touch their own data)
We need to build 3 things:
    1. A JWT Filter: 'checkpoint' that runs before every request to check token valid or not.
        * The reason why we extract and put 'email' into SecurityContextHolder instead of passing as a variable to Controller
            -> avoid "Parameter Pollution", put 'email' to SecurityContextHolder like Global Variable
    2. Todo Service: handle logic CRUD
    3. Todo Controller: specific endpoints

Phase 6: Testing
    1. Unit test
        * Goal: Test 1 single class in isolation
        * Use mockito  & Junit toolkit
        * Pattern : AAA
            + Arrange : Set up the data and tell Mockito how to behave
            + Act : Call the actual method want to test
            + Assert : Check if the result is what expected
        * Annotation:
            + @Mock: create "empty shell" objects of our repositories and security classes. these objects
            have no logic, they only do what we tell them to do during the test
            + @InjectMocks: This creates a real instance of 'TodoService' and automatically "plugs in"
            the mocked 'TodoRepository' and 'UserRepository' into it. and also ensure 'TodoService' isn't talking
            to the actual PostgreSQL database
        * Pattern explanation:
            + Arrange Phase: Setting the Scene
                - For example, User A ( Attacker): ID:1, email:user_a@example.com; User B(Victim): ID:2,email: everything@example.com
                    we create a task (todo object) with Id:100 and set owner is User B.
                - Mocking Security Context: in the real application we have JWTAuthenticationFilter, it sets the user in
                the "Security Context". In the unit test , it does not run -> do it manually
                    1. Tell the 'SecurityContext' return 'authentication' mock
                    2. tell 'authentication' mock return String 'user_a@test.com' when asked for the "Principal" (the logged in identity)
                    3. Finally, tell the 'userRepository' to return User A when the service searches for that email.
                - Mocking Repository: tell the 'todoRepository' that when someone asks for ID 100, it should return the Todo belongs
                to the User B.
            + Act & Arrange Phase: Testing the logic
                - assertThrows(RuntimeException.class,...). This is a special JUnit method. It says "I'm going to call 'deleteTodo
                and I expect it to fail with a RuntimeException". If the code succeeds or throws a different error, the test fails
                - Service call, not match -> throw exception
                - assertEquals(...) verify error message the user sees is exactly what we intended, preventing weird error logs(what
                we doesn't expect to see)
            + Verify phase: Final Security check
                + 'verify(todoRepository, never()).delete(any())'-> "Safety Net". Check delete method on DB repo was called or not ?
                    - if logic was broken and the code accidentally deleted the task before checking the ID, this line would let
                    the test fail
                    - So that , we has proven that not only an error was thrown , but also no data was harmed in the process.
